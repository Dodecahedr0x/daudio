# daudio-ui (Vizia Widgets) — Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Add a shared `daudio-ui` crate (theme + a `ParamControl` widget + an editor helper), wire an `editor()` seam into the SDK (`DaudioEffect` + the `#[daudio_plugin]` macro), and give the filter a real custom themed editor — replacing nih-plug's generic auto-UI.

**Architecture:** `daudio-ui` builds on `nih_plug_vizia` (nih-plug's Vizia editor integration). It provides a themed stylesheet, a `ParamControl` view wrapping nih_plug_vizia's built-in `ParamSlider`, and an `editor(...)` helper that hides the `create_vizia_editor` + `ViziaState` + `Lens`/`Data` boilerplate. The SDK gains an optional `editor()` method on `DaudioEffect`; the macro generates `Plugin::editor()` delegating to it, so effect plugins opt in by overriding one method and everything else is unchanged.

**Tech Stack:** Rust nightly, nih-plug + nih_plug_vizia (pinned rev `f36931f7af4646065488a9845d8f8c2f95252c23`), Vizia (via nih_plug_vizia).

**CANONICAL REFERENCE:** nih-plug's own `plugins/examples/gain_gui_vizia/src/editor.rs` in the pinned repo does exactly this (ViziaState, a `#[derive(Lens)]` Data struct, `create_vizia_editor`, a stylesheet, `ParamSlider`). READ IT before writing UI code and match its API usage for the pinned rev — nih_plug_vizia's exact signatures (stylesheet application, `ViziaTheming`, `create_vizia_editor`) may differ from this plan's sketch; the example is authoritative.

**Step 1: `param_control.rs`** — a labeled slider. Model it on how the example composes a `Label` + `ParamSlider` in a `VStack`:
```rust
use nih_plug::prelude::Param;
use nih_plug_vizia::vizia::prelude::*;
use nih_plug_vizia::widgets::ParamSlider;

/// A labeled parameter control: a caption above nih_plug_vizia's ParamSlider,
/// styled by the daudio theme.
pub struct ParamControl;

impl ParamControl {
    pub fn new<L, Params, P, FMap>(cx: &mut Context, label: &str, params: L, params_to_param: FMap)
    where
        L: Lens<Target = Params> + Clone,
        Params: 'static,
        P: Param + 'static,
        FMap: Fn(&Params) -> &P + Copy + 'static,
    {
        VStack::new(cx, |cx| {
            Label::new(cx, label).class("daudio-label");
            ParamSlider::new(cx, params, params_to_param);
        })
        .class("daudio-control");
    }
}
```
> The generic bounds must match nih_plug_vizia's `ParamSlider::new` signature EXACTLY for the pinned rev (the `Lens` + map-fn shape). Copy the bound pattern from the example's usage. If the exact generics are awkward, it is acceptable for `ParamControl::new` to mirror `ParamSlider::new`'s signature verbatim.

**Step 2: `editor.rs`** — hide the `create_vizia_editor` + Data/Lens boilerplate:
```rust
use nih_plug::prelude::Editor;
use nih_plug_vizia::vizia::prelude::*;
use nih_plug_vizia::{create_vizia_editor, ViziaState, ViziaTheming};
use std::sync::Arc;

/// Lens data wrapper so widgets can bind to `Arc<Params>`.
#[derive(Lens)]
struct DaudioData<Params: 'static> {
    params: Arc<Params>,
}

impl<Params: 'static> Model for DaudioData<Params> {}

/// Create a default editor window state (width x height).
pub fn editor_state(width: u32, height: u32) -> Arc<ViziaState> {
    ViziaState::new(move || (width, height))
}

/// Build a daudio editor: applies the theme, sets up param binding, and runs `content`.
pub fn create_editor<Params, C>(
    state: Arc<ViziaState>,
    params: Arc<Params>,
    content: C,
) -> Option<Box<dyn Editor>>
where
    Params: 'static + Send + Sync,
    C: Fn(&mut Context) + Copy + Send + Sync + 'static,
{
    create_vizia_editor(state, ViziaTheming::Custom, move |cx, _| {
        crate::theme::apply_theme(cx);
        DaudioData { params: params.clone() }.build(cx);
        content(cx);
    })
}
```
> Verify EVERY nih_plug_vizia symbol here against the pinned rev: `create_vizia_editor`'s exact signature, `ViziaState::new`, `ViziaTheming` variants, the closure arity `(cx, gui_cx)`, and that `#[derive(Lens)]` + `Model` is the right binding pattern. The example is the source of truth. If `DaudioData` must be accessed by widgets (for the lens), export it or expose a lens accessor as the example does (`Data::params`). Adjust the `content` closure signature if widgets need the lens passed in rather than reading it from context.
